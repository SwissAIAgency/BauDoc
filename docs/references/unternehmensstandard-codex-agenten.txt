1. Ziel des Standards
Dieser Leitfaden beschreibt, wie ein Unternehmen mehrere Softwareprojekte mit Codex strukturiert aufsetzt. Ziel ist nicht, dass ein Agent unkontrolliert eine komplette Software baut, sondern dass jedes Projekt nach einem wiederholbaren Standard entsteht.

Kontrolle
Jede Änderung hat Task, Agent, Dateigrenzen, Akzeptanzkriterien und QA-Prüfung.

Wiederholbarkeit
Projektdefinition, Skills, UI-Standards und Konventionen werden in Markdown-Dateien hinterlegt.

Teamfähigkeit
Alle Agenten arbeiten über ein Repository, Branches, Pull Requests, Review-Regeln und Dokumentation.

Wichtig: Codex soll nicht einfach „alles bauen“. Codex soll nach einem Unternehmensstandard planen, delegieren, implementieren, prüfen und dokumentieren.
2. Grundprinzip der Agentenarbeit
Die Standardlogik lautet:

Idee
→ Projektdefinition
→ Repository initialisieren
→ AGENTS.md und Unternehmensregeln hinterlegen
→ Orchestrator-Agent starten
→ Architektur und Datenmodell planen
→ Tasks definieren
→ spezialisierte Agenten umsetzen lassen
→ QA, Security und Datenschutz prüfen
→ Pull Request
→ Review
→ Merge
→ Dokumentation aktualisieren
Die wichtigste Regel
Setze nicht direkt alles um.
Arbeite als Orchestrator:
analysieren, zerlegen, begrenzen, delegieren, prüfen und dokumentieren.
3. Empfohlenes Agentenmodell
Für DB-basierte Webapplikationen mit Unternehmensstandard empfehle ich folgendes Agentensystem:

Agent	Aufgabe	Darf ändern?	Wann einsetzen?
Orchestrator-Agent	Koordiniert Projekt, Tasks, Agenten, Reihenfolge und Reviews.	Nur Dokumentation/Planung, wenn erlaubt.	Immer.
Architektur-Agent	Plant technische Struktur, Module, Schnittstellen und Risiken.	Nein, primär Analyse und Dokumentation.	Vor Projektstart und bei Architekturänderungen.
Backend-Agent	API, Services, Businesslogik, Validierung, Backendtests.	Ja, nur Backend-Dateien.	Bei API- und Logik-Tasks.
Frontend-/UI-Agent	UI, Komponenten, Layouts, Responsiveness, Interaktionen.	Ja, nur Frontend-Dateien.	Bei UI-Tasks.
DB-Agent	Datenmodell, Migrationen, Relationen, Indizes, Constraints.	Ja, nur DB-/Schema-Dateien nach Freigabe.	Bei jeder DB-Änderung.
Datenschutz- & Compliance-Agent	Schweizer DSG/nDSG, DSGVO-Risiken, Datenflüsse, Löschung, Consent.	Nein, prüft und definiert Anforderungen.	Bei personenbezogenen Daten, Tracking, KI, Voice, RAG, Uploads.
DevOps-/Security-Agent	Docker, Deployments, CI/CD, Env, Secrets, Hosting, Security.	Ja, nur Infrastrukturdateien.	Bei Deployment, Ports, Secrets, CI/CD.
QA-/Review-Agent	Prüft Änderungen gegen Auftrag, Tests, Standards und Regressionen.	Nein, ausser Reparatur ausdrücklich erlaubt.	Nach jeder Implementierung.
Unternehmensregel: Ein Agent darf nie eigenmächtig ausserhalb seiner Rolle arbeiten. Der Backend-Agent ändert keine DB-Struktur ohne DB-Agent. Der Frontend-Agent ändert keine API-Verträge. Der DevOps-Agent ändert keine Businesslogik.
4. Repository-Workflow für Zusammenarbeit der Agenten
Sobald mehrere Agenten oder mehrere Codex-Projekte entstehen, braucht jedes Projekt ein Repository mit klaren Branch- und Review-Regeln. Das Repository ist die gemeinsame Wahrheit für Code, Standards, Dokumentation, Skills und Entscheidungen.

Empfohlenes Branch-Modell
main
├── develop
│   ├── feature/backend-customer-api
│   ├── feature/frontend-customer-ui
│   ├── feature/db-customer-model
│   ├── fix/qa-login-validation
│   └── chore/devops-docker-config
Branch-Regeln
main ist geschützt und enthält nur freigegebenen Code.
develop ist Integrationsbranch für geprüfte Features.
Jeder Task bekommt einen eigenen Branch.
Kein Agent arbeitet direkt auf main.
Kein Merge ohne QA-Bericht.
DB-Migrationen brauchen DB-Agent + QA + Datenschutzprüfung, falls personenbezogene Daten betroffen sind.
UI-Änderungen brauchen Abgleich mit Brand- und UI-Standards.

7. MD-Regeln und Unternehmenskonventionen
Unternehmensstandards dürfen nicht nur im Prompt stehen. Sie müssen als dauerhafte Markdown-Dateien im Repository liegen.

AGENTS.md
# AGENTS.md

## Unternehmensstandard

Dieses Projekt folgt dem standardisierten Entwicklungsmodell des Unternehmens.

Alle Agenten müssen lesen:
- docs/00_projektdefinition.md
- docs/04_restriktionen.md
- docs/13_ui_brand_guidelines.md
- docs/14_repository_workflow.md
- relevante Skills aus /skills

## Grundregeln

- Arbeite nur am aktuellen Task.
- Ändere nur erlaubte Dateien.
- Keine grossen Refactorings ohne Freigabe.
- Keine neuen Dependencies ohne Begründung.
- Keine bestehenden Funktionen entfernen.
- Keine globalen Konfigurationsänderungen ohne Auftrag.
- Keine Ports ändern ohne DevOps-/Security-Agent.
- Keine UI-Änderungen bei Backend-Tasks.
- Keine Backend-Änderungen bei UI-Tasks.
- Keine DB-Änderung ohne DB-Agent.
- Keine personenbezogenen Daten ohne Datenschutz- & Compliance-Agent.
- Keine Secrets im Code.
- Kein Merge ohne QA-Review.

## Rollenpflicht

Jeder Task muss einem Agenten zugeordnet werden:
- Orchestrator-Agent
- Architektur-Agent
- Backend-Agent
- Frontend-/UI-Agent
- DB-Agent
- Datenschutz- & Compliance-Agent
- DevOps-/Security-Agent
- QA-/Review-Agent

## Definition of Done

Ein Task ist fertig, wenn:
- Ziel erfüllt
- erlaubte Dateien eingehalten
- Tests/Checks dokumentiert
- QA geprüft
- Datenschutz/Security geprüft, falls relevant
- UI-/Brand-Standards eingehalten, falls relevant
- Dokumentation aktualisiert
- PR bereit oder gemerged

# skills/safe-feature-implementation.md

## Ziel

Ein Feature klein, sicher und prüfbar umsetzen.

## Ablauf

1. Task lesen.
2. AGENTS.md lesen.
3. Erlaubte Dateien prüfen.
4. Gesperrte Bereiche prüfen.
5. Minimalen Implementierungsplan erstellen.
6. Nur notwendige Änderungen durchführen.

- Grosse Refactorings
- Nebenfeatures
- globale Änderungen
- neue Dependencies ohne Freigabe
